Common Errors & Solutions

A comprehensive collection of known errors, their causes, and fixes for Droidspaces.

Installation & Setup Errors

GrapheneOS Not Supported

Error: Droidspaces fails to run on GrapheneOS.

Cause: GrapheneOS blocks critical syscalls needed for namespace isolation and containerization, making it impossible to run a userspace runtime like Droidspaces even with root access.

Solution: GrapheneOS is not supported. Use a different ROM or device.

SuSFS Conflicts

Error: Containers fail to boot when SuSFS is installed.

Cause: SuSFS interferes with Droidspaces' filesystem operations.

Solution: Ensure "HIDE SUS MOUNTS FOR ALL PROCESSES" is disabled in your SuSFS4KSU settings. Do not report bugs when using SuSFS.

"Required key not available" (ENOKEY)

Symptoms: Container crashes or filesystem operations fail with "Required key not available" errors. Most common on Android devices with File-Based Encryption (FBE).

Cause: systemd services inside the container attempt to create new session keyrings, causing the process to lose access to Android's FBE encryption keys.

Affected kernels: 3.18, 4.4, 4.9, 4.14, 4.19

Solution: - Update Droidspaces to v4.2.4+ - Run droidspaces check to verify seccomp support - Ensure CONFIG_SECCOMP=y and CONFIG_SECCOMP_FILTER=y in your kernel - Switch to rootfs.img mode (recommended on Android) - Advanced: Decrypt /data by editing fstab in boot partitions

Mount Errors on Kernel 4.14

Symptoms: First container start after stopping fails with a mount error, but retry succeeds.

Cause: On kernel 4.14, loop device cleanup is asynchronous. After unmounting a rootfs image, the loop device may not be fully released.

Solution: Droidspaces v4.2.3+ handles this automatically with a 3-attempt retry loop. If still experiencing issues: - Update to the latest version - Wait a few seconds between stop and start - Use sync before restarting

OverlayFS Not Supported (f2fs)

Symptoms: --volatile mode fails with an error about OverlayFS not supporting f2fs.

Cause: Most Android devices use f2fs for /data. OverlayFS on many Android kernels does not support f2fs as a lower directory.

Solution: Use a rootfs image instead of a directory:

# Will fail on f2fs:
droidspaces --rootfs=/data/rootfs --volatile start

# Will work:
droidspaces --name=test --rootfs-img=/data/rootfs.img --volatile start

Runtime Errors

Container Name Conflicts

Symptoms: Starting a container fails because a container with the same name is already running, or PID file conflicts occur.

Solution: 1. Check running containers: droidspaces show 2. Clean up stale state: droidspaces scan 3. Use a different name: droidspaces --name=mycontainer-2 start

Systemd Hangs on Older Kernels

Symptoms: systemd hangs or becomes unresponsive on legacy kernels (3.18, 4.4, 4.9, 4.14, 4.19).

Cause: systemd's service sandboxing triggers a race condition in the kernel's VFS grab_super path.

Solution: Enable the Deadlock Shield in the app or use --block-nested-namespaces in CLI, hard reboot, and try again.

Container Won't Stop

Symptoms: droidspaces stop takes more than 15 seconds and eventually fails.

Cause: Same as systemd hangs on older kernels — VFS deadlock.

Solution: Enable the Deadlock Shield, hard reboot, and try again.

Modern Distros Fail on Legacy Kernels

Symptoms: Arch Linux, Fedora, or openSUSE fail to boot with "Unsupported Kernel" or hang during systemctl commands.

Cause: Modern systemd (v258+) requires kernel features not present in kernels older than 5.4.

Solution: - Use distributions with OpenRC, runit, or s6 - Use Ubuntu 22.04/24.04/25.04 or Debian 12/13 (older systemd) - Use Alpine Linux (OpenRC-based)

Networking Errors

Ping Fails with "socket: permission denied"

Cause: Android Paranoid Networking — the kernel restricts socket creation to specific GIDs (AID_INET 3003, AID_NET_RAW 3004, AID_NET_ADMIN 3005).

Solution: - Kernel level: Set CONFIG_ANDROID_PARANOID_NETWORK=n - Userland: Use our pre-patched rootfs tarballs

DNS / Name Resolution Issues

Symptoms: Internet works (can ping IPs), but domain names fail to resolve.

Cause: Some ISPs block common DNS servers like 8.8.8.8 and 1.1.1.1.

Solution: Use your ISP's DNS servers:

dumpsys connectivity | sed 's/}}/\n/g' | grep 'InterfaceName: wlan0' | grep -o 'DnsAddresses: \[[^]]*\]' | grep -o '/[0-9]*\.[0-9]*\.[0-9]*\.[0-9]*' | tr -d '/'

Add the results as DNS servers in the container configuration.

WiFi/Mobile Data Disconnects

Symptoms: WiFi or mobile data stops working on the host during container start/stop.

Cause: systemd-networkd inside the container conflicts with Android's network management.

Solution: - Mask systemd-networkd inside the container:

sudo systemctl mask systemd-networkd
  • Use isolated NAT mode (--net=nat)

Wi-Fi Power Save Causes Sluggish Networking

Symptoms: Slow networking when the device screen is off.

Solution: Create a watchdog container to keep Wi-Fi power save disabled:

# Install iw
apk add iw  # Alpine
apt install iw  # Debian/Ubuntu

# Create watchdog script at /usr/local/bin/wifi-watchdog.sh
while true; do
    if iw dev wlan0 get power_save 2>/dev/null | grep -q "on"; then
        iw dev wlan0 set power_save off
    fi
    sleep 60
done

Set up as an OpenRC or systemd service. Requires host networking mode (--net=host).

Filesystem & Storage Errors

SELinux-Induced Rootfs Corruption (Directory Mode)

Symptoms: Symbolic link sizes changing unexpectedly, library load failures, or random binary crashes.

Cause: Android's SELinux policy assigns a generic context to /data/local/Droidspaces/Containers, interfering with filesystem operations in directory mode.

Solution: Switch to rootfs.img mode (--rootfs-img=/path/to/rootfs.img). This encapsulates the filesystem inside an ext4 image, preventing SELinux from relabeling individual files.

Rootfs Image I/O Errors on Android

Symptoms: Loop-mounting a rootfs image silently fails.

Cause: The SELinux context of the .img file prevents the loop driver from performing I/O.

Solution: Droidspaces v4.3.0+ applies the correct SELinux context automatically. For older versions:

chcon u:object_r:vold_data_file:s0 /path/to/rootfs.img

Reclaiming Storage After Deletion (Sparse Image)

Symptoms: rootfs.img doesn't shrink after deleting files inside the container.

Cause: The host filesystem can't detect when blocks are freed inside the ext4 image.

Solution: Run fstrim inside the container:

sudo fstrim -av

Systemd Service Errors

Systemd Service Sandboxing Conflicts (Legacy Kernels)

Symptoms: Services like redis, mysql, or apache fail with exit-code or status=226/NAMESPACE.

Cause: Modern systemd sandboxing directives conflict with Droidspaces' Adaptive Seccomp Shield on legacy kernels.

Solution: Create a service override:

sudo systemctl edit

Add:

[Service]
PrivateTmp=no
PrivateDevices=no
ProtectSystem=no
ProtectHome=no
RestrictNamespaces=no
MemoryDenyWriteExecute=no
NoNewPrivileges=no
CapabilityBoundingSet=

Then:

sudo systemctl daemon-reload
sudo systemctl restart

Getting Help

If your issue isn't listed here:

  1. Run droidspaces check and note any failures
  2. Check container logs: droidspaces --name=mycontainer run journalctl -n 100
  3. Try foreground mode: droidspaces --name=mycontainer --rootfs=/path/to/rootfs --foreground start
  4. Join the Telegram channel for community support
  5. Open an issue on the GitHub repository
